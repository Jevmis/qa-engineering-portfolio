# Bug Management

> Identifying, documenting, prioritising, tracking, and verifying software defects throughout the development lifecycle.

---

# Overview

Effective defect management is essential for delivering reliable software.

A good bug report does more than describe an issue—it provides developers with the information they need to reproduce, understand, and resolve the problem efficiently.

---

# My Bug Reporting Process

Every defect I report includes:

- Clear Title
- Environment
- Module
- Build Version
- Severity
- Priority
- Preconditions
- Steps to Reproduce
- Expected Result
- Actual Result
- Supporting Evidence
- Additional Notes

---

# Severity vs Priority

I distinguish between:

## Severity

The technical impact of the defect.

Examples:

- Critical
- High
- Medium
- Low

---

## Priority

How urgently the defect should be resolved.

Examples:

- Immediate
- High
- Medium
- Low

---

# Defect Lifecycle

A typical workflow includes:

1. New
2. Assigned
3. In Progress
4. Fixed
5. Ready for Testing
6. Verified
7. Closed

If necessary:

- Reopened

---

# Good Bug Reports

Effective bug reports should be:

- Reproducible
- Concise
- Accurate
- Evidence-based
- Easy to understand

Supporting evidence may include:

- Screenshots
- Screen recordings
- Logs
- API responses
- SQL queries

---

# Collaboration

Bug management is a collaborative activity involving:

- QA Engineers
- Developers
- Business Analysts
- Product Owners
- Project Managers

Clear communication helps teams resolve issues more efficiently.
